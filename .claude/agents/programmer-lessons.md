- [PR #8] Todo projeto deve ter .github/workflows/ci.yml com npm ci, typecheck, lint e build desde o início.
- [PR #8] Hardcode de cor em componente visual é BLOCKER — sempre usar var(--token) do design system, inclusive em hover/active states.
- [PR #8] Botões de ícone (IconButton, IconBtn) sem :focus-visible são BLOCKER de acessibilidade — sempre incluir outline: 2px solid var(--accent) com outline-offset: 2px.
- [PR #8] Variantes secondary e ghost do Button sem :focus-visible são BLOCKER — todo variante que remove outline no base precisa de substituto (box-shadow ring) nos blocos :focus-visible de cada variante.
- [PR #8] Campo de busca (SearchInput) com outline: none sem :focus-within no container é BLOCKER de acessibilidade — sempre adicionar :focus-within com border-color e box-shadow ring no wrapper SearchBox.
- [PR #8] tokens.css órfão causa confusão sobre fonte de verdade — tokens de z-index devem estar em design.css junto com os demais tokens; nunca criar arquivo de tokens paralelo sem importá-lo em lugar algum.
- [PR #12] Border-width literal em componente base é BLOCKER — sempre que existir valor numérico de borda (1px, 1.5px, 2px, 2.5px), criar token semântico em tokens.css (`--border-thin/medium/thick/thicker`) e referenciar via `var(--…)`; ausência do token na fonte não é justificativa para aceitar literal, é sinal de que o token precisa ser adicionado.
- [PR #12] Magic numbers em estados interativos (`translateY(1px)` no `:active`) são BLOCKER — criar token dedicado (`--press-offset`) em vez de valor literal; qualquer literal numérico em CSS de componente base deve ter token correspondente.
- [PR #42] Ao introduzir tema dark, auditar TODOS os assets binários (SVG/PNG) usados em componentes globais (Sidebar, Topbar, layouts) — assets com fill fixo escuro/claro precisam de variante oposta selecionada via `useTheme().resolvedTheme`; deixar como "fora de escopo" é BLOCKER quando a issue inclui critério "componentes existentes renderizam corretamente em ambos os temas". Verifique contraste WCAG AA (≥3:1 para UI graphics non-text) de cada asset com `--bg-surface` de cada tema antes de abrir PR.
- [Sonar] Qualquer issue nova (Bug, Vulnerability, Security Hotspot ou Code Smell) introduzida pelo PR no SonarCloud em arquivo tocado pelo diff é BLOCKER, independentemente da severidade — antes de abrir PR, aguardar o check `SonarCloud Code Analysis` ou rodar análise local e zerar todas as issues novas; issues pré-existentes em `development` não contam, mas qualquer regressão no diff atual é veto automático do reviewer.
- [PR #112] Mudança de copy fora do escopo declarado da issue é BLOCKER — issue #111 era exclusivamente sobre aspect-ratio da logo, mas alterei `<BrandTitle>Entrar no painel</BrandTitle>` para `Acessar no painel`, derrubando o teste `LoginPage.test.tsx` e o CI; ao tocar uma página existente, releia a issue e o diff antes de salvar para garantir que NENHUMA string/copy fora do escopo foi alterada, e antes do push rode `npm test` no container para confirmar 236/236 reais — nunca reportar contagem de testes sem ter executado a suíte completa localmente.
- [PR #119] Fail-fast no boot lendo `import.meta.env.VITE_*` quebra o CI se a env não estiver definida no GitHub Actions — o `.env` é gitignored, e `loadEnv` em `vite.config.ts` não encontra o valor em runners limpos. Sempre que adicionar `throw new Error(...)` em `src/shared/api/index.ts` (ou similar) por env ausente, **simultaneamente** definir um valor sintético em `vite.config.ts > test.env.<VAR>` para o Vitest injetar em `import.meta.env` durante os testes; senão qualquer test file que importe `@/shared/auth` (transitivamente puxa o singleton `apiClient`) falha no parse antes do primeiro `it()`. Validar localmente removendo o `.env` e rodando `npm test` simula CI fielmente.
- [PR #123] Duplicação em testes do `AuthProvider`/`guards` derrubou Quality Gate do Sonar pela 2ª vez (>3% New Code Duplication) — repetir o erro do PR #119. Antes de pushar PRs grandes em `tests/shared/auth/*`, escanear blocos repetidos com olho clínico: (a) `renderHook+waitFor(isLoading=false)` é fixture (`setupAuth(client)`), não copy-paste; (b) `seedSession+login+logout` é fixture (`setupLoggedInProvider`); (c) cenários com a mesma estrutura mudando 1-2 mocks são `it.each`, não `it` separados; (d) type guards quase idênticos em arquivos diferentes (`isValidPermissionsResponse` vs `isValidCachedPermissions`) precisam de helper compartilhado em `types.ts`/módulo dedicado, não duplicação. **Critério prático antes do push:** se o mesmo trecho de 10+ linhas aparece em 2+ testes/arquivos, refatore antes — Sonar conta blocos de 10+ linhas como duplicação independente da intenção.
- [PR #127] **3ª recorrência** do Sonar New Code Duplication (4.8% > 3%) em testes de `SystemsPage`. Extrair `makeSystem`/`makePagedResponse`/`createSystemsClientStub` em `__helpers__/` foi correto **mas insuficiente**. O bloco `vi.mock('@/shared/auth', …)` (~12 linhas) foi duplicado entre `SystemsPage.test.tsx` e `SystemsPage.create.test.tsx` (resolvido via `buildAuthMock`), e o boilerplate "abrir modal → preencher form → submeter" se repetia em 8 testes (resolvido extraindo `openCreateModal`/`fillNewSystemForm`/`submitNewSystemForm`). Cenários `400/401/403/network` que diferiam só em 1-2 mocks também ficaram inline em vez de `it.each`. **Regra prática:** ao tocar 2+ arquivos de teste do mesmo recurso, leia-os lado a lado **antes do push** procurando trechos ≥10 linhas idênticas — e converta describe-de-cenários-similares em `it.each` com tabela.
- [PR #128] **4ª recorrência** do Sonar New Code Duplication (7.0%) — desta vez em **código de produção** (não só testes). `NewSystemModal.handleSubmit` e `EditSystemModal.handleSubmit` compartilhavam ~25 linhas no `catch (error: unknown) { if (isApiError(error)) { if (status === 409) ... if (=== 400) ... if (=== 401 || === 403) ... } }`, divergindo apenas em literais ("criar"/"atualizar", "um sistema"/"outro sistema"). Sonar tokeniza isso como bloco duplicado **e** o aninhamento dispara Cognitive Complexity > 15 (BLOCKER paralelo). Solução: extrair função pura `classifySubmitError(error, copy: SubmitErrorCopy): SubmitErrorAction` em `systemFormShared.ts` retornando ação discriminada (`{kind: 'conflict' | 'bad-request' | 'not-found' | 'toast' | 'unhandled'}`); cada modal vira `switch (action.kind)` curto chamando seu side-effect. **Regra cumulativa para os PRs futuros do CRUD (Roles, Users, Clients, Permissions):** desde o **primeiro PR do recurso**, projetar o módulo `<recurso>FormShared.ts` com (i) tipos de form, (ii) validação client-side, (iii) parsing de `ValidationProblemDetails`, (iv) `classifySubmitError` pra mapeamento de erros — não esperar o segundo modal aparecer e depois refatorar; o "vamos extrair quando duplicar" custou 4 BLOCKERs de Sonar. Ao tocar 2+ arquivos similares (criar/editar, list/show, etc.), o programmer **deve** fazer um diff side-by-side antes do push e refatorar qualquer bloco ≥10 linhas idêntico, não confiar em ter extraído helpers parcialmente.
- [PR #134] **5ª recorrência** do Sonar New Code Duplication (4.7%) — desta vez **entre dois recursos diferentes** (sistemas vs rotas), não entre create/edit do mesmo recurso. Mesmo tendo extraído `systemFormShared.ts`/`routeFormShared.ts` paralelos cada um com `classify*SubmitError`, o Sonar tokeniza o **corpo** das duas funções como bloco duplicado de 26 linhas (mesmo switch 409/400/404/401/403/unhandled). Casos detectados pela análise: (i) `useSystemForm`/`useRouteForm` repetiam ~19 linhas de handlers `handleNameChange/handleCodeChange/handleDescriptionChange`; (ii) `classifySubmitError`/`classifyRouteSubmitError` tinham 26 linhas idênticas; (iii) `SystemsPage`/`RoutesPage` repetiam ~28 linhas de boilerplate de paginação (`computeTotalPages` + `useMemo` + `isFirstPage/isLastPage` + `handlePrevPage/handleNextPage`). **Regra prática reforçada:** módulos `<recurso>FormShared.ts` paralelos são insuficientes — quando o **corpo** das funções (não só a forma) é idêntico entre recursos, extrair em `src/shared/forms/` (helpers genéricos parametrizados por `TField`) ou `src/hooks/` (hooks de UI). Antes de abrir PR de qualquer novo recurso de CRUD, escanear `src/pages/<recurso>/use<Recurso>Form.ts` e `<recurso>FormShared.ts` contra os equivalentes já existentes — qualquer bloco ≥10 linhas idêntico (independente de literais que diferem) deve ser refatorado para um helper genérico (ex.: `useFieldChangeHandlers`, `classifyApiSubmitError`, `usePaginationControls`).
